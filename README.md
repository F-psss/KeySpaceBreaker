# KeySpaceBreaker

Система для распределённого взлома классических шифров методом полного перебора
ключей. Зашифрованный текст разбивается на диапазоны ключей («юниты»), которые
параллельно перебираются множеством воркеров. Координатор раздаёт работу, собирает
результаты и выбирает наилучший вариант расшифровки по эвристической оценке текста.

## Возможности

- Перебор ключей шифров Цезаря и Виженера.
- Два режима для Виженера: полный перебор (`brute`) и ускоренный (`fast`).
- Горизонтальное масштабирование: произвольное число воркеров.
- Поддержка кластера координаторов с автоматическим выбором лидера (Bully election).
- Восстановление из чекпоинтов: прогресс расшифровки сохраняется на диск и
  восстанавливается при перезапуске координатора.

## Архитектура

Система состоит из трёх типов узлов.

**Координатор** принимает задачу от клиента, разбивает пространство ключей на юниты
и раздаёт их воркерам. Собирает результаты, отслеживает таймауты невыполненных
юнитов и периодически сохраняет прогресс в контрольную точку. По завершении всех
юнитов отправляет наилучший результат клиенту.

**Воркер** подключается к координатору, получает юнит (диапазон ключей и
зашифрованный текст), перебирает ключи, оценивает кандидатов и возвращает лучший
результат по своему диапазону.

**Клиент** отправляет координатору задачу (шифр, текст, параметры) и ожидает
итоговый результат.

### Отказоустойчивость

Координаторы образуют группу и общаются между собой по отдельному соединению
(peer-to-peer, полносвязная топология). В каждый момент времени один координатор
является основным (Primary), остальные — резервными (Backup).

- **Выбор роли.** При старте координаторы определяют роли детерминированно: основным
  становится узел с наименьшим идентификатором среди живых. Первые
  `INITIAL_ROLE_GRACE_SEC` (15 с) после запуска — начальная задержка, в течение которой
  узлы ждут остальных перед объявлением себя основными.
- **Обнаружение отказа.** Основной координатор каждые `HEARTBEAT_INTERVAL_SEC` (2 с)
  рассылает резервным heartbeat (`PEER_PING`). Резервные обнаруживают его смерть по
  обрыву соединения или по тайм-ауту `HEARTBEAT_TIMEOUT_SEC` (10 с).
- **Перевыборы.** При смерти основного координатора резервные запускают выборы
  (по алгоритму Bully election): узел рассылает `PEER_ELECTION` всем с меньшим `id`; если за
  `ELECTION_TIMEOUT_SEC` (3 с) никто не ответил `PEER_ALIVE`, объявляет себя
  основным и рассылает `PEER_COORDINATOR`, иначе отступает и остаётся резервным.
- **Обслуживание клиентов/воркеров.** Резервные координаторы не принимают
  подключения воркеров и клиентов — их порты для этих подключений закрыты. Только
  основной координатор обслуживает работу. При смене роли порты открываются или
  закрываются автоматически.
- **Переподключение.** Воркеры и клиент при запуске получают список адресов всех
  координаторов и при потере связи перебирают их, пока не найдут текущего основного.
  Клиент при этом повторно отправляет задачу.

> **Ограничение.** Реализация не использует кворумный консенсус. При сетевом
> разделении возможна ситуация двух основных координаторов;
> после восстановления связи конфликт разрешается по правилу
> «наименьший идентификатор побеждает», но прогресс, наработанный проигравшей
> половиной за время разделения, не консолидируется.

### Протокол

Узлы обмениваются сообщениями в формате JSON поверх TCP. Каждое сообщение содержит
тип (`REQUEST` / `RESPONSE` / `ERROR`), действие и полезную нагрузку. Кадрирование —
32-байтовый заголовок с длиной тела + тело; максимальный размер сообщения — 1000 КБ.

Служебные действия между координаторами: `PEER_HELLO`, `PEER_PING`, `PEER_ELECTION`,
`PEER_ALIVE`, `PEER_COORDINATOR`, `PEER_CHECKPOINT`. Прикладные действия: `DECRYPT`,
`STATUS`, `PING`, `QUIT`.

## Зависимости

- Компилятор с поддержкой C++20.
- [CMake](https://cmake.org/) 3.10 или новее.
- [Asio](https://think-async.com/Asio/) 1.36.0 (standalone) — асинхронный
  ввод-вывод; поставляется в `include/asio-1.36.0/`.
- [CLI11](https://github.com/CLIUtils/CLI11) — разбор аргументов командной строки;
  поставляется в `include/CLI/`.
- [nlohmann/json](https://github.com/nlohmann/json) 3.12.0 — сериализация JSON;
  поставляется в `include/nlohmann/`.
- OpenSSL (`libssl`, `libcrypto`) — используется для Base64-кодирования передаваемых
  текстов через OpenSSL BIO API (`include/BASE64.hpp`).

Сторонние header-only библиотеки поставляются вместе с исходным кодом в каталоге
`include/`. OpenSSL устанавливается системным пакетным менеджером:

```bash
# Debian/Ubuntu
sudo apt install libssl-dev

# Fedora
sudo dnf install openssl-devel

# macOS (Homebrew)
brew install openssl
```

## Сборка

```bash
mkdir build && cd build
cmake ..
make
```

После сборки в каталоге `build/` появятся три исполняемых файла:

- `run_coord` — координатор;
- `run_worker` — воркер;
- `main` — клиент.

## Запуск

Ниже — пример развёртывания с тремя координаторами, одним воркером и одним клиентом
на локальной машине. Каждый координатор использует три порта: для воркеров, для
клиента и для связи с другими координаторами.

### 1. Координаторы

Запустите каждый координатор в отдельном терминале. Параметр `--peers` — список
адресов остальных координаторов (для связи между собой).

```bash
./run_coord --id 1 --worker-port 1025 --client-port 1026 --peer-port 1027 \
            --peers localhost:2027,localhost:3027 --checkpoint coord1.json

./run_coord --id 2 --worker-port 2025 --client-port 2026 --peer-port 2027 \
            --peers localhost:1027,localhost:3027 --checkpoint coord2.json

./run_coord --id 3 --worker-port 3025 --client-port 3026 --peer-port 3027 \
            --peers localhost:1027,localhost:2027 --checkpoint coord3.json
```

| Флаг | Описание |
|------|----------|
| `--id` | Уникальный идентификатор координатора (целое число; меньший = выше приоритет при выборах). |
| `--worker-port` | Порт для подключения воркеров. |
| `--client-port` | Порт для подключения клиента. |
| `--peer-port` | Порт для связи с другими координаторами. |
| `--peers` | Список адресов `host:port` (peer-порты) остальных координаторов через запятую. |
| `--checkpoint` | Путь к файлу контрольной точки (по умолчанию: `coordinator_checkpoint.json`). |

После старта основным (Primary) станет координатор с наименьшим `--id` среди
запущенных; он откроет порты для воркеров и клиента. Запуск одиночного координатора
(без `--peers`) тоже поддерживается — он сразу становится основным.

### 2. Воркеры

Воркеру передаётся список адресов всех координаторов (их worker-порты) и путь к
словарю. Воркер сам найдёт основного координатора и переподключится при сбое.

```bash
./run_worker --coordinators localhost:1025,localhost:2025,localhost:3025 \
             --dict words.txt
```

| Флаг | Описание |
|------|----------|
| `--coordinators` | Список адресов `host:port` (worker-порты) координаторов через запятую. |
| `--dict` | Путь к файлу словаря (см. ниже). |

Можно запускать несколько воркеров одновременно — они подключатся к одному и тому же
основному координатору и разделят работу.

### 3. Клиент

Клиенту передаётся список адресов координаторов (их client-порты) и параметры задачи.

```bash
# Шифр Цезаря
./main --cipher caesar --input "Khoor Zruog" \
       --servers localhost:1026,localhost:2026,localhost:3026

# Шифр Виженера (полный перебор), текст из файла
./main --cipher vigenere --mode brute --key-length 3 --input-file encrypted.txt \
       --servers localhost:1026,localhost:2026,localhost:3026

# Шифр Виженера (быстрый режим), длина ключа определяется автоматически
./main --cipher vigenere --mode fast --input-file encrypted.txt \
       --servers localhost:1026,localhost:2026,localhost:3026
```

| Флаг | Описание |
|------|----------|
| `--cipher` | Тип шифра: `caesar` или `vigenere`. |
| `--mode` | Режим для Виженера: `brute` (полный перебор) или `fast` (ускоренный). Обязателен для Виженера. |
| `--key-length` | Длина ключа Виженера. Если не задана, оценивается автоматически. |
| `--input` | Зашифрованный текст строкой. |
| `--input-file` | Путь к файлу с зашифрованным текстом (взаимоисключающе с `--input`). |
| `--servers` | Список адресов `host:port` (client-порты) координаторов через запятую. |
| `--noise` | Уровень «шума» для оценки (0..1): при `noise > 0` оценка комбинирует частотный анализ и словарную эвристику. |
| `--output-file` | Путь для сохранения результата (опционально). |

Клиент подключается к текущему основному координатору, отправляет задачу и ожидает
результат. Если основной координатор выходит из строя во время расшифровки, клиент
переподключается к новому основному и автоматически повторяет задачу.

## Шифры и режимы

**Цезарь** — моноалфавитный сдвиговый шифр. Пространство ключей мало (26 сдвигов),
перебирается мгновенно.

**Виженер** — полиалфавитный шифр с ключевым словом. Пространство ключей растёт
экспоненциально с длиной ключа (26^N).

- Режим `brute` перебирает все ключи заданной длины, расшифровывая полный текст для
  каждого кандидата и оценивая его по частотам букв английского алфавита (χ²-статистика).
  При ненулевом `--noise` оценка дополнительно учитывает словарь: `score = freq_score × (1 - noise²) - dict_score × noise × 100`.
- Режим `fast` вычисляет оценку без расшифровки — по частотам символов в каждом
  потоке шифртекста (χ² по ожидаемым частотам английского). Работает значительно
  быстрее, но может уступить в точности на коротких текстах. Если длина ключа не
  задана явно, она оценивается автоматически методом Касиски с уточнением по индексу
  совпадений (IC).

## Оценка качества расшифровки

Воркер ранжирует кандидатов по χ²-отклонению частот букв от эталонного распределения
английского языка. Меньше χ² — лучше: идеально расшифрованный текст должен
воспроизводить ожидаемые частоты (E: 12,7%, T: 9,1%, A: 8,2%, …).

Словарь используется как дополнительный сигнал при `noise > 0`: чем больше слов из
словаря встречается в кандидате, тем выше его оценка по словарю.

## Словарь

Файл словаря (`words.txt`) — обычный текстовый файл со списком слов (по одному в
строке), используемый для оценки качества расшифровки.
Передаётся воркеру через флаг `--dict`.

## Контрольные точки

Координатор сохраняет прогресс каждые `CHECKPOINT_EVERY_N_UNITS` (5) завершённых
юнитов в JSON-файл (путь задаётся флагом `--checkpoint`). При перезапуске координатор
загружает контрольную точку и продолжает с сохранённого места, если задача совпадает
(идентификатор задачи — длина и начало зашифрованного текста).

Чтобы начать задачу заново, удалите файл контрольной точки перед запуском.
